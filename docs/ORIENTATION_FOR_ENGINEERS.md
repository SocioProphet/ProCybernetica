# Orientation for Engineers

ProCybernetica is a blueprint-codification repository.

The captured corpus defines an architecture for cybernetic control, epistemic governance, replay, lawful learning, secure coordination, and collective stabilization.

## Engineering posture

Build from the captured blueprint.

Do not begin with generic agent runtime assumptions.

Do not collapse the architecture into a prompt wrapper, vector store, workflow engine, or service mesh. Those are implementation substrates. The blueprint is broader and stricter.

## Core objects to understand

### Fractal Node

Every meaningful component is a node: repository, agent, process, planner, executor, sensor, gateway, operator, robot, host, or service.

A node has identity, lifecycle, interfaces, memory, world model, value judgment, behavior generation, execution, learning hooks, coordination, and observability.

### Authority tree and information graph

Authority and information are different planes.

Information can move laterally. Authority must remain typed, scoped, auditable, and replayable.

### Repository as active node

A repository is not a bucket. It is a governed memory organ with admission, retention, conflict, replication, provenance, and promotion law.

### Proposal versus promotion

A model, planner, prompt, extractor, schema, policy, or artifact may be proposed cheaply.

Promotion is a constitutional act. It requires evidence, replay, policy pass, attestation, and rollback posture.

### Genesis and Inception

Genesis produces reusable seeds.

Inception binds a seed to actor, project, trust domain, runtime, region, mission, policy, memory, and provider context to produce a live twin.

### Replay as constitutional evidence

Replay is not debug-only. It is how the system reconstructs causality, evaluates alternatives, and learns without normalizing accidents as truth.

## Implementation sequence

Follow this order:

1. Reconcile schemas and profiles across source captures.
2. Freeze v0 canonical envelope names and enums.
3. Build schema validation and fixture pack.
4. Build lifecycle supervisor and transition records.
5. Build replay/evaluation/promotion path.
6. Build claim/event/provenance path from Book XI.
7. Add mesh, identity, policy bundle, attestation, and operator workflow slices.
8. Add scoring/dashboard ingestion only after public/private review.

Reconcile them against:

- Volume II architecture specification;
- Volume III executable specification pack;

- Book XI implementation practicum.

## Definition of engineering progress

Good progress means the repo becomes more self-contained, more faithful to the blueprint, more testable, and less ambiguous.

Bad progress means adding impressive code before the contracts are reconciled.

## Public safety

This is a public repository. Do not commit secrets, customer data, private telemetry, live operational logs, or non-public evidence. Use synthetic fixtures or sanitized examples.
