## Why ontology belongs in the cybernetics prelude

Ontology is not an optional semantic-web add-on. It is the formal discipline that makes a knowledge system:

- shareable;
- queryable;
- alignable;
- governable across contexts.

Earlier books already depend on ontology:

- Book II needs typed claims, schema and ontology references, and promotion law;
- Book VI needs semantic commons and perspective-bearing relational structure;
- Book VII needs a Fractal Node Contract whose surfaces are semantically typed, not only syntactically typed.

Ontology is the reference semantic model that gives integration, interoperability, analytics, and data access their common context.

## Syntax, semantics, and models

Syntax specifies which strings and graph fragments are well-formed.

Semantics specifies how those forms are interpreted, when an interpretation satisfies axioms, and how truth conditions are induced over a model.

Without that distinction, ontology collapses into naming or mystified rhetoric about meaning.

OWL 2 is captured as formalized vocabularies defined through relationships and axioms, with profiles tuned for computational tradeoffs.

RDF provides graph data model.

RDFS adds basic typing and class/property hierarchy.

OWL 2 adds stronger axiomatics and profile families.

SPARQL provides query.

SHACL provides graph-shape validation.

Together, this stack turns graph data into typed, queryable, constrainable semantic state.

## Ontology languages and tractable profiles

Ontology languages are engineering languages.

- RDF is permissive and graph-first.
- RDFS gives lightweight schema.
- OWL 2 gives description-logic basis and profile families such as EL, QL, and RL.

The hard lane needs fragments that remain computationally tame enough to commit truth and drive side effects.

If the commit path depends on unbounded reasoning, promotion, replay, and assurance become fragile.

This makes bounded fragments a constitutional safety concern, not only an implementation preference.

## Query, validation, and promotion

SPARQL and SHACL are gatekeepers between representation and commitment.

SPARQL turns semantic data into operational retrieval and transformation medium.

SHACL turns semantic commitments into graph-shape constraints checked before promotion.

Soft-lane processors may retrieve, align, rank, and suggest.

Hard-lane processors must validate.

A claim does not become canonical because a large model emits it or a lexical matcher scores it highly. It becomes canonical only after schema, ontology, and policy gates pass.

## Ontology-research architecture in Drive

The Drive ontology papers present an architecture pattern:

1. existing sources are wrapped by conversion/adaptation layer;
2. reference ontology layer mediates access and harmonization;
3. central graph/graphstore plus service layer exposes search, navigation, analysis, versioning, user/admin services;
4. application layer consumes common semantic model rather than heterogeneous source APIs.

This is a teaching example for the knowledge plane:

- sources;
- adaptation;
- ontology;
- graph middleplane;
- services;
- application use.

## Ontology-based data integration and mediation

The clinical integration paper teaches ontology as executable mediation law.

Source ontology, target ontology, and mapping ontology are kept distinct.

Mappings are used to generate and organize integration work such as SQL extraction templates.

A mediator cannot safely align heterogeneous systems if correspondences are hidden in procedural code or untracked prompts.

Approximate alignment via lexical similarity, structural matching, semantic similarity, or model extraction begins in the soft lane. Promoted mappings enter the hard lane only after evidence, validation, and review.

## SSN / QUDT / GeoSPARQL pattern

The smarter-industry materials show reference ontologies harmonized together:

- SSN for stimulus-sensor-observation and sensing capability;
- GeoSPARQL for spatial features, geometry, and query idioms;
- QUDT for quantity kinds, units, dimensions, and quantity values;
- asset ontologies for ownership, registry, and configuration context.

Lessons:

1. reference ontologies are stronger together when a domain needs observation, measurement, location, and operational context;
2. identity is delicate; `owl:sameAs` is often too strong;
3. bridging classes can avoid false identity collapse;
4. ontologies gain power when wired into query and service layers.

## SocioProphet ontology operations

The source maps local repositories to operational roles:

- `ontogenesis`: auditable, policy-gated, supply-chain-traceable ontology distributions with repeatable builds, deterministic hashing, append-only ledgers, COSE signing, SHACL promotion gates, and layered architecture across upper, middle, lower, domain, and platform modules;
- `ontology-engineering`: educational runway for what ontologies are, how to build/evaluate them, how description logic, knowledge graphs, and ontology languages fit;
- `ontology-access-kit`: operational API runway across files, SPARQL endpoints, APIs, validation, matching, annotation, and subgraphs;
- `semantic-serdes`: boundary-contract runway preserving plane, truth class, time model, merge model, provenance, governance status, and replayability.

## Action ontology and process ontology

The source extends beyond noun-oriented ontologies.

Heylighen's action ontology treats actions and agents as primitive and explains complex phenomena as self-organizing networks of interacting agents.

A cybernetic system cannot be governed by static noun taxonomy alone.

It needs ontologies of:

- observation;
- action;
- capability;
- role;
- transition;
- side effect;
- organizational state;
- transformation.

Ontology and cybernetics become the same discipline viewed from different levels.

## Computability and the tame commit path

The more expressive a representation becomes, the harder safe global reasoning becomes.

Not every meaningful problem has a computationally safe realization.

Expressivity has costs.

The commit path must stay tame:

- tractable profiles;
- bounded rule fragments;
- shape validation;
- replayable processors;
- policy gates.

Embeddings, ranking, lexical matchers, semantic alignment candidates, large-model extraction, and probabilistic mappings belong in the soft lane until promoted.

This is a computability-aware constitutional boundary.
